# Build Notes

- 2026-07-10: Starting build. Empty dir, no git repo yet. Node v24.17.0, npm 11.17.0.
- 2026-07-10: `lucide-svelte` is deprecated in favor of `@lucide/svelte` — swapped immediately, not in the SPEC but worth noting for future svelte builds.
- 2026-07-10: `npx sv create . --template minimal` refused to run in a non-empty dir (SPEC.md + NOTES.md already present) and needed the interactive "Directory not empty?" prompt answered via piped `y`.
- 2026-07-10: Vitest 4 renamed `test.workspace` to `test.projects` — the older `workspace` key silently type-errored under `svelte-check` (`'workspace' does not exist in type 'InlineConfig'`), not caught by oxlint/oxfmt, only by `svelte-check`. Fixed by renaming the key and switching `vite.config.ts`'s `defineConfig` import from `'vite'` to `'vitest/config'` so the `test` field type-checks at all.
- 2026-07-10: oxlint (via the ultracite preset) is much stricter than expected out of the box — flagged `func-style` (prefers arrow consts over function declarations), `sort-keys` on every object literal, `no-array-sort` (wants `toSorted`), `prefer-ternary`, `no-unused-expressions` (broke on the common Svelte 5 `$effect(() => { void someRune; ... })` dependency-tracking idiom until wrapped in `void`), and unicode-flag/named-capture-group requirements on every regex literal. Delegated the bulk mechanical cleanup to a subagent since it touched ~10 files; had to manually restore a few `let` bindings the subagent's earlier prefer-const pass had incorrectly converted to `const` for values only reassigned via `bind:this`/event handlers (oxlint's static analysis can't see template-side reassignment).
- 2026-07-10: First CI run failed on the e2e "404" test. Initially misdiagnosed as PokeAPI network flakiness (added a Playwright route mock + retries) since it passed 100% locally every time. Root cause was actually that the workflow built once with `BASE_PATH=/pokedex-sum-sm2` and then ran Playwright against that base-prefixed build via `npm run preview`, but the tests navigate with root-relative paths (matching local dev, where `BASE_PATH` is unset) — so _every_ route silently hit a wrong/missing path under the base prefix, not just the 404 test coincidentally. The other 6 tests happened to still pass because IntersectionObserver-driven loads and client-side routing don't hard-depend on the initial URL matching in the same way. Fixed by splitting the workflow into two build steps: one with no base for e2e, one with the Pages base path only for the deploy artifact. Lesson: always reproduce CI's exact env vars locally before trusting a "flaky test" diagnosis — I burned two failed CI runs and ~10 minutes before building locally with `BASE_PATH` set and reproducing instantly.
- 2026-07-10: Lighthouse accessibility caught something none of my manual review passes did: several Pokémon type badge colors (grass #7ac74c, fire #ee8130, water #6390f0, bug #a6b91a, etc.) fail WCAG AA contrast (4.5:1) against the hardcoded white badge text — only 5 of 18 types (fighting, poison, ghost, dragon, dark) are actually dark enough to need white text. Computed per-type luminance/contrast in a throwaway Python script and added a `typeTextColor()` helper. Went from Accessibility 96 → 100.
- 2026-07-10: Build complete. Live at https://azagatti.github.io/pokedex-sum-sm2/. Lighthouse: Accessibility 100, Best Practices 100, SEO 100, Agentic Browsing 100 (Performance not scored in this Lighthouse mode, but trace-based LCP ~650ms / CLS 0.00 on both local and production, comfortably in the "good" range).
